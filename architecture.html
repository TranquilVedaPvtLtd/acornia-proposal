Key Decisions

Why This Stack?

Every architectural choice has a clear rationale tied to the platform's specific needs — financial data compliance, cost efficiency, and developer velocity.

Decision Choice Alternatives Rationale
Database Supabase PostgreSQL PlanetScale, Neon, Firebase Auth + Realtime + RLS + Edge Functions in one platform. ACID compliance critical for financial data. No vendor lock-in — standard PostgreSQL underneath.
Bulk Storage Cloudflare R2 AWS S3, Supabase Storage only Zero egress fees — critical for report downloads (InvestWell data can be large and frequently accessed). S3-compatible, so easy to migrate if needed.
Frontend Next.js 15 Remix, SvelteKit, Vite SPA Largest React ecosystem. Server components reduce bundle size. API routes eliminate need for separate backend. Vercel hosting with zero-config deploys.
Auth Supabase Auth Clerk, NextAuth, Auth0 Native integration with RLS — database permissions enforced at SQL level, not just API level. No additional vendor cost. JWT-based, stateless.
Realtime Supabase Realtime Pusher, Ably, Socket.io Included in Supabase plan — no extra cost. WebSocket-based, listens to database changes directly. Perfect for notification bell and Kanban updates.
Verification Cashfree API Surepass, CVL KRA direct Single vendor for both PAN and bank verification. Well-documented API, sandbox environment for testing, reasonable pricing, good uptime.
Report Automation Playwright Puppeteer, Selenium Best cross-browser support, typed API, auto-waits for elements. Actively maintained by Microsoft. Can run in lightweight containers.
Hosting Vercel Cloudflare Pages, Railway Best Next.js support (created by same team). Edge functions, preview deploys per PR, analytics built-in. Zero-config deployment from GitHub.

Security Architecture

7 Layers of Protection

Defence in depth — every layer independently protects the system. A breach at one layer does not compromise the others.

1Cloudflare CDN + WAF + DDoS ProtectionEdge-level filtering
2HTTPS Everywhere (TLS 1.3)Encryption in transit
3Supabase Auth (JWT Sessions)Identity verification
4Row Level Security (PostgreSQL RLS)Database-level access control
5API Input Validation (Zod Schemas)Data integrity enforcement
6Encrypted Secrets (Supabase Vault)API keys, credentials protected
7Complete Audit Trail (Every Mutation)Full accountability

Sensitive Data Handling

  • Passwords: bcrypt hashed (Supabase Auth)
  • API keys: AES-256 encrypted (Supabase Vault)
  • PAN numbers: Displayed masked (last 4 only)
  • Bank accounts: Last 4 digits stored only
  • Platform credentials: Encrypted at rest
  • Meeting audio: Signed URLs (time-limited access)

Backup & Recovery

  • Database: Supabase daily automated backups
  • Point-in-time recovery available
  • R2 Storage: 99.999999999% durability
  • Code: GitHub with branch protection
  • Secrets: Separate from code, rotatable
  • Uptime target: 99.5%
